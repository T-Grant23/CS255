# CS255
Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?
The purpose of the project is to create an online platform for new drivers to sign up for scheduled driving lessons as well as access practice materials for DMV driving tests. The client is DriverPass, and on the admin end of the platform DriverPass wants to be able to download user data and reports, reset passwords, block lower admin access, track, view and enter data related to users and trainers, add/remove packages offered by the company, and connect to the DMV website. On the user end of the platform, they want to allow users to sign up for accounts, choose lesson packages, enter user information, schedule driving lessons, access practice materials, and view lesson details and progress. 

What did you do particularly well?
I think I performed an in-depth analysis into the client's system requirements that allowed me to provide the client with invaluable information. Details such as functional/nonfunctional requirements, proposed user interface, assupmtions, and limitations lay out a roadmap for a well-designed application.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
If I had to revise one component of my document it would be the technical requirements of the system design document because it could've been more extensive.

How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?
Interview notes were analyzed to determine functional/nonfuncational requirements, design constraints, user needs, project timelines, and UML diagrams. It's important to consider user needs because those needs can be translated into required system functionality.

How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?
In order to deliver my software design to the client, I initially began by gathering functional and nonfunctional system requirements. Design constraints were later determined and the collected information was used to illustrate the system architecture. UML diagrams provided a blueprint of the proposed system, and technical requirements were explained to the client.
